# API 设计规范

## 设计原则

### 就近定义

数据结构应当与使用它的 API 模块同文件声明，不应为复用而单独抽离。只有当同一数据结构被三个及以上不同业务模块共享时，才考虑提取为公共类型。

### 类型即文档

接口类型名称应清晰表达业务语义，避免无前缀的通用命名（如 `Item`、`Response`）。建议使用 `业务实体 + 语义后缀` 的命名模式，如 `Account`、`AccountUpdate`、`OpportunityDetail`。

### 拒绝冗余

已有的类型不可在同文件中重复定义，功能相近的结构应通过可选字段或联合类型合并，而非创建多个近似类型。

## 目录组织

按业务领域划分 API 模块，各模块职责单一：

| 模块 | 职责范围 |
|------|----------|
| 认证模块 | 登录、注册、Token 管理、登出 |
| 账号模块 | 账号 CRUD、评价模板、扫码登录 |
| 商品模块 | 商品列表、上下架、分组 |
| 选品模块 | 关键词、监控商品/商家、选品统计 |
| 发布模块 | 商机、素材、图片上传 |
| 设置模块 | AI 配置、通知渠道等 |

新建 API 时应归入已有模块或创建新模块，而非在现有文件中追加不相关的接口。公共工具函数应收敛到统一位置，供各模块复用。`lib/utils/api.ts` 的 `fetchApi` 是项目中唯一的 HTTP 请求入口，所有 API 模块通过可选参数（`params`/`skipAuth`/`baseUrl`/`credentials_`）定制行为，禁止在模块中自行封装 fetch。

## 类型定义规范

### 命名层级分明

类型命名应反映其用途，应用一致的后缀加以区分：

- 完整实体 — 表示完整的业务对象
- 更新入参 — 表示修改时的输入结构
- 创建入参 — 表示新建时的输入结构
- 列表精简版 — 用于列表页展示的精简字段集
- 详情展开结构 — 用于详情页的完整字段集

### DTO 与前端类型区分

后端直传的数据结构建议标注后缀（如 `XxxDTO`），前端自行计算或转换的衍生类型建议使用独立名称（如 `XxxView`），避免两者混用。

### 通用响应类型

通用操作结果反馈类型用于操作成功/失败+消息的返回，不应嵌入业务数据字段。如需返回业务数据，应定义独立的响应类型。

## 避免的模式

- 在多个文件中定义语义相同的重复类型
- 使用过于宽泛的类型（如 `Record<string, unknown>`）作为业务数据字段的类型
- 在 API 文件中混入与接口无关的工具函数
- 导出仅供内部使用的中间类型
- 各模块各自重复定义相同的请求工具函数
