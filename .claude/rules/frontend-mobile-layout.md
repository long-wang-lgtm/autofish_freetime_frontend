# 移动端布局设计规范

> 纯粹的布局原则，指导后续移动端 UI 设计决策。不包含具体页面或组件的实现细节。

## 核心原则

### 1. 顶部空间优先级

移动端视口顶部是最珍贵的空间。元素的排版优先级从高到低：

1. **导航**（tab 切换 / 页面标题）— 最高优先，占据第一行
2. **搜索 / 筛选** — 紧接导航下方
3. **操作入口**（个人中心 / 设置 / 管理）— 退到角落或收入抽屉

PC 端独占一行的顶栏，在移动端应收缩为角落小控件或收入侧边抽屉，释放顶部空间给导航和内容。

### 2. 导航层级分明

- **一级导航**：应用级页面切换（如侧边栏抽屉、底部 tab bar），始终可触达
- **二级导航**：页面内 tab 切换，`sticky top-0` 贴顶，滚动时保持可见
- **三级筛选**：子标签 / 分类 pill，在二级导航下方，随内容滚动

每级导航的视觉重量逐级递减（字号、颜色浓度、间距）。

### 3. z-index 分层

移动端交互层级多（遮罩、抽屉、浮动按钮、贴顶导航），必须分层明确：

```
z-30  ← 遮罩层 / Modal / Toast
z-25  ← 侧边抽屉 / 下拉面板
z-20  ← 浮动控件（FAB / 角落按钮）
z-10  ← sticky 贴顶元素（tab 导航栏）
z-0   ← 页面正文内容
```

同一个 z 层内的元素不应互相遮挡。如有竞争空间的情况（如贴顶 tab 行与右上角浮动控件），应通过 margin/padding 预留空间，而非提高 z-index 硬遮挡。

### 4. 内容区从视口顶部开始

移动端 `main` 容器不应有顶部 padding。顶部间距由页面内第一个可见元素（导航栏 / 标题）自行提供。这样：
- 有 tab 的页面：tab 行紧贴视口顶部
- 无 tab 的页面：标题/工具栏自带 `pt-*`
- 不会出现无意义的空白带

### 5. 横向溢出：滑动优先于换行

移动端宽度有限，当 tab / pill / chip 数量超出屏幕宽度时：
- 使用 `overflow-x-auto` 横向滑动，而非换行
- 隐藏滚动条（`hide-scrollbar`），保持视觉干净
- 右侧如有浮动元素，预留空间防止遮挡

### 6. 触摸热区最小 44×44px

移动端可点击元素的最小触摸区域为 44×44px（`min-h-11 min-w-11`），无论视觉上多小。CSS `padding` 可补偿视觉尺寸与实际热区的差距。

### 7. PC 与移动端用同一份 HTML 结构

通过 Tailwind 响应式前缀（`lg:` / `max-lg:`）切换样式，而非 `if (isMobile)` 渲染两套不同的 DOM。只有功能逻辑有本质差异时才用 JS 分支。

## 反模式

- 移动端顶栏继续占一整行——浪费顶部黄金空间
- z-index 随意设 50 / 100 / 999——应使用统一的分层体系
- sticky 元素没有背景色——下方内容透过
- 移动端 main 保留 PC 的顶部 padding——出现无意义空白
- tab 过多时换行——应横向滑动
- 点击目标小于 44×44px——触摸操作困难
- JS 判断 `isMobile` 渲染两套完全不同的 DOM——应优先用响应式 CSS
